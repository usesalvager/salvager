# Releasing

Releases are automated. Push a `v*` tag and the `CI & Release` workflow does the rest:

```sh
git tag v1.2.0
git push origin v1.2.0
```

That triggers, in order:

1. **ci** — build, vet, test.
2. **create-release** — a GitHub Release with autogenerated notes.
3. **release** — static binaries (`CGO_ENABLED=0`, `-trimpath`) for
   linux/darwin × amd64/arm64, each with a `.sha256`, attached to the release.
4. **bump-tap** — regenerates `Formula/salvager.rb` in
   [`usesalvager/homebrew-tap`](https://github.com/usesalvager/homebrew-tap) with
   the new version and the checksums the release just produced, and **pushes it
   directly** (no PR). The checksums come from the same release assets users
   download, so there is no separate source to keep in sync.

## One-time setup for the tap bump

`bump-tap` is gated on a secret and skips cleanly if it is absent, so the rest of
the release still works before this is configured.

1. **Create the tap repo** `usesalvager/homebrew-tap` (must be named
   `homebrew-tap` for `brew install usesalvager/tap/salvager` to resolve) with a
   `Formula/salvager.rb` (see the formula in that repo / generated by `bump-tap`).
2. **Create a fine-grained PAT** scoped to **only** `usesalvager/homebrew-tap`,
   with **Contents: Read and write** and nothing else, and an expiration date.
   The default `GITHUB_TOKEN` cannot be used — it is scoped to this repo and does
   not cross repositories.
3. **Add it as a secret** named `TAP_PUSH_TOKEN` in this repo
   (Settings → Secrets and variables → Actions).

Rotate the PAT before it expires; until `TAP_PUSH_TOKEN` is valid, `bump-tap`
no-ops and the formula must be bumped manually.

## Pinning / verifying an install

The install script and formula both consume the release assets directly:

```sh
SALVAGER_VERSION=v1.2.0 \
  curl -fsSL https://raw.githubusercontent.com/usesalvager/salvager/main/install.sh | sh
```

The installer verifies each binary's `.sha256` before installing; a mismatch
aborts with nothing written.
